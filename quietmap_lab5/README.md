# Лабораторные работы 1–5: языковой процессор для объявления кортежа Elixir

## Автор

Андрей.  
Дисциплина: проектирование языковых процессоров / компиляторы.

## Краткое описание проекта

Проект представляет собой графическое приложение-текстовый редактор, в которое поэтапно встроены модули языкового процессора. В качестве индивидуального варианта используется упрощённая учебная конструкция объявления кортежа в стиле Elixir:

```elixir
tuple = {:ok, "Hello", 42};
```

Важно: анализируется не весь язык Elixir, а только заданная конструкция: идентификатор, знак присваивания, кортеж в фигурных скобках и завершающий символ `;`.

Основные возможности программы:

- создание, открытие и сохранение текстовых файлов;
- редактирование текста в многострочном редакторе;
- лексический анализ входного текста;
- синтаксический анализ объявления кортежа;
- восстановление после синтаксических ошибок методом Айронса;
- поиск подстрок по трём регулярным выражениям;
- построение AST;
- выполнение семантических проверок;
- вывод результатов анализа в таблицы с позициями ошибок и найденных фрагментов.

## Используемые технологии

- Язык программирования: Python 3.
- GUI-фреймворк: PySide6.
- Основной файл программы: `src/main.py`.
- Файл зависимостей: `requirements.txt`.
- Сборка исполняемого файла: PyInstaller через GitHub Actions или локально.

## Сборка и запуск

### Запуск из исходного кода

```bash
python -m pip install -r requirements.txt
python src/main.py
```

### Локальная сборка исполняемого файла

```bash
python -m pip install -r requirements.txt
python -m pip install pyinstaller
pyinstaller --onefile --windowed src/main.py
```

После сборки файл находится:

- Windows: `dist/main.exe`;
- Linux/macOS: `dist/main`.

В репозитории также есть workflow `.github/workflows/build-win-linux.yml`, который собирает артефакты для Windows и Linux.

# Лабораторная работа 2. Лексический анализатор

## Цель работы

Изучить назначение и принципы работы лексического анализатора в структуре компилятора, спроектировать конечный автомат и реализовать сканер для выделения лексем из входного текста.

## Вариант задания

Вариант: объявление кортежа в стиле Elixir.

Общий вид корректной строки:

```elixir
identifier = {value1, value2, ...};
```

Пример:

```elixir
tuple = {:ok, "Hello", 42};
```

Допустимые элементы кортежа:

- атомы: `:ok`, `:error`, `:user`;
- строки: `"Hello"`;
- целые числа без знака: `42`;
- идентификаторы: `user`, `tuple`, `value1`;
- вложенные кортежи: `{:ok, {:user, 1}}`.

## Таблица лексем

| Код | Тип лексемы | Пример |
|---:|---|---|
| 1 | целое без знака | `42` |
| 2 | идентификатор | `tuple` |
| 3 | атом | `:ok` |
| 4 | строковый литерал | `"Hello"` |
| 10 | оператор присваивания | `=` |
| 11 | разделитель: запятая | `,` |
| 12 | открывающая скобка кортежа | `{` |
| 13 | закрывающая скобка кортежа | `}` |
| 14 | пробел / табуляция | пробел |
| 15 | перенос строки | `\n` |
| 16 | конец оператора | `;` |
| 99 | ошибка | недопустимый символ |

## Диаграмма состояний лексического анализатора

![Диаграмма состояний лексера](docs/state-diagram-elixir.svg)

Краткое описание работы автомата:

1. Сканер последовательно просматривает входную строку слева направо.
2. Если встречается буква нижнего регистра или `_`, начинается распознавание идентификатора.
3. Если встречается `:`, далее ожидается имя атома.
4. Если встречается цифра, распознаётся целое число без знака.
5. Если встречается `"`, распознаётся строковый литерал до закрывающей кавычки.
6. Символы `{`, `}`, `,`, `=`, `;` распознаются как отдельные лексемы.
7. Любой другой символ считается ошибкой, для него фиксируется строка и позиция.

## Пример корректного лексического анализа

Входная строка:

```elixir
tuple = {:ok, "Hello", 42};
```

Ожидаемый результат:

| Код | Тип | Лексема | Местоположение |
|---:|---|---|---|
| 2 | идентификатор | `tuple` | строка 1, 1–5 |
| 14 | пробел / табуляция | пробел | строка 1, 6–6 |
| 10 | оператор присваивания | `=` | строка 1, 7–7 |
| 14 | пробел / табуляция | пробел | строка 1, 8–8 |
| 12 | открывающая скобка кортежа | `{` | строка 1, 9–9 |
| 3 | атом | `:ok` | строка 1, 10–12 |
| 11 | запятая | `,` | строка 1, 13–13 |
| 14 | пробел / табуляция | пробел | строка 1, 14–14 |
| 4 | строковый литерал | `"Hello"` | строка 1, 15–21 |
| 11 | запятая | `,` | строка 1, 22–22 |
| 14 | пробел / табуляция | пробел | строка 1, 23–23 |
| 1 | целое без знака | `42` | строка 1, 24–25 |
| 13 | закрывающая скобка кортежа | `}` | строка 1, 26–26 |
| 16 | конец оператора | `;` | строка 1, 27–27 |

## Пример лексической ошибки

Входная строка:

```elixir
tuple = {:ok, @};
```

Символ `@` не относится к допустимым лексемам, поэтому программа выводит ошибку вида:

| Код | Тип | Лексема | Местоположение |
|---:|---|---|---|
| 99 | ошибка: недопустимый символ `@` | `@` | строка 1, позиция 15 |

# Лабораторная работа 3. Синтаксический анализатор

## Цель работы

Изучить назначение синтаксического анализатора, разработать грамматику для заданной конструкции, реализовать парсер и нейтрализацию синтаксических ошибок методом Айронса.

## Грамматика

```text
<program>   -> <nl>* <stmt_list> <nl>*
<stmt_list> -> <stmt> (<nl>+ <stmt>)*
<stmt>      -> <id> '=' <tuple> ';'
<tuple>     -> '{' <elems_opt> '}'
<elems_opt> -> ε | <elems>
<elems>     -> <value> (',' <value>)*
<value>     -> <id> | <atom> | <int> | <string> | <tuple>
<nl>        -> '\n'
```

## Классификация грамматики

Грамматика является контекстно-свободной, то есть относится к типу 2 по классификации Хомского. Левая часть каждого правила содержит один нетерминал, а правая часть задаёт последовательность терминалов и нетерминалов.

## Метод анализа

В программе используется метод рекурсивного спуска. Основные процедуры:

- `parse_program` — разбор всей программы;
- `parse_statement` — разбор одного объявления;
- `parse_tuple` — разбор кортежа;
- `parse_value` — разбор элемента кортежа.

Схема синтаксического анализа:

![Схема синтаксического анализа](docs/parser-flow.svg)

## Диагностика и нейтрализация ошибок

Нейтрализация ошибок реализована по идее метода Айронса: анализатор не завершает работу после первой ошибки, а пытается продолжить разбор с ближайшей безопасной позиции.

В программе используются следующие действия восстановления:

- виртуальная вставка пропущенного символа `=` после идентификатора;
- виртуальная вставка пропущенной запятой между элементами кортежа;
- сообщение о пропущенном `;` в конце объявления;
- синхронизация по опорным токенам: `NEWLINE`, `IDENTIFIER`, `,`, `}` и `;`.

## Тестовые примеры

### Корректная строка

```elixir
tuple = {:ok, "Hello", 42};
```

Ожидаемый результат: синтаксических ошибок нет.

### Пропущена запятая

```elixir
tuple = {:ok "Hello", 42};
```

Ожидаемая ошибка:

| Неверный фрагмент | Местоположение | Описание |
|---|---|---|
| `"Hello"` | строка 1, позиция 14 | Ожидалась `,` между элементами кортежа |

### Не закрыт кортеж и отсутствует `;`

```elixir
tuple = {:ok, "Hello", 42
```

Ожидаемые ошибки:

| Неверный фрагмент | Местоположение | Описание |
|---|---|---|
| `\n` или `EOF` | строка 1 / конец ввода | Ожидалась `}` в конце кортежа |
| `EOF` | конец ввода | Ожидался `;` в конце объявления |

# Лабораторная работа 4. Поиск подстрок по регулярным выражениям

## Цель работы

Изучить регулярные выражения и применить их для поиска подстрок в тексте. Интегрировать поиск в графический интерфейс программы.

## Постановка задачи

В программу добавлен режим поиска по трём заданиям. Результаты выводятся в таблицу со столбцами:

- найденная подстрока;
- начальная позиция;
- длина.

При выборе строки результата соответствующий фрагмент подсвечивается в редакторе.

Схема поиска:

![Схема поиска по регулярным выражениям](docs/regex-flow.svg)

## Задание 1. SSN

Формат: `XXX-XX-XXXX`, где `X` — цифра.

Регулярное выражение:

```regex
(?<!\d)\d{3}-\d{2}-\d{4}(?!\d)
```

Пояснение:

- `\d{3}` — три цифры;
- `-` — дефис;
- `\d{2}` — две цифры;
- `\d{4}` — четыре цифры;
- `(?<!\d)` и `(?!\d)` запрещают лишние цифры слева и справа.

Найдёт:

```text
123-45-6789
```

Не найдёт:

```text
123456789
12-345-6789
123-45-67890
```

## Задание 2. Размер файла

Формат: число + единица измерения `KB`, `MB`, `GB` или `TB`.

Регулярное выражение:

```regex
(?<![\w.])\d+(?:\.\d+)?(?:KB|MB|GB|TB)\b
```

Пояснение:

- `\d+` — целая часть числа;
- `(?:\.\d+)?` — необязательная дробная часть через точку;
- `(?:KB|MB|GB|TB)` — допустимая единица измерения;
- `\b` — граница слова.

Найдёт:

```text
10KB
1.5GB
256MB
```

Не найдёт:

```text
12PB
GB
1,5GB
```

## Задание 3. VIN

Формат: 17 символов, разрешены цифры и латинские буквы, кроме `I`, `O`, `Q`.

Регулярное выражение:

```regex
\b[A-HJ-NPR-Z0-9]{17}\b
```

В программе для VIN дополнительно реализован поиск конечным автоматом: `RegexSearchEngine._find_vin_by_automaton()`.

Найдёт:

```text
1HGCM82633A004352
JH4TB2H26CC000000
```

Не найдёт:

```text
1HGCM82633A00I352
1HGCM82633A00435
X1HGCM82633A004352Y
```

## Общий тестовый текст

```text
User SSN: 123-45-6789
Backup size: 1.5GB and 256MB
VIN: 1HGCM82633A004352
Invalid: 12-345-6789, 12PB, 1HGCM82633A00I352
```

Ожидаемые совпадения:

| Режим | Найденные подстроки |
|---|---|
| SSN | `123-45-6789` |
| Размеры файлов | `1.5GB`, `256MB` |
| VIN | `1HGCM82633A004352` |

# Лабораторная работа 5. AST и семантический анализ

## Цель работы

Изучить назначение семантического анализатора, построение AST и проверку контекстно-зависимых условий для заданной конструкции.

## Постановка задачи

Необходимо развить синтаксический анализатор до семантического:

- построить AST для корректной синтаксической конструкции;
- проверить контекстно-зависимые условия;
- вывести дерево AST и список семантических ошибок.

## Пример корректной строки

```elixir
tuple = {:ok, {:user, 1}, "hi"};
```

## Структура AST

Для входной строки выше программа строит дерево следующего вида:

```text
ProgramNode
└── StatementNode
    ├── name: "tuple"
    └── value: Tuple
        ├── node: :{}
        ├── meta: []
        └── args
            ├── :ok
            ├── Tuple
            │   ├── node: :{}
            │   ├── meta: []
            │   └── args
            │       ├── :user
            │       └── 1
            └── "hi"
```

Типы узлов AST:

| Узел | Назначение |
|---|---|
| `ProgramNode` | корень дерева, содержит список объявлений |
| `StatementNode` | одно объявление вида `id = tuple;` |
| `TupleNode` | кортеж и список его элементов |
| `AtomNode` | атом вида `:ok` |
| `IntegerNode` | целое число без знака |
| `StringNode` | строковый литерал |
| `IdentifierNode` | использование идентификатора внутри кортежа |

## Контекстно-зависимые условия

В программе реализованы следующие проверки:

| Правило | Что проверяется | Пример ошибки |
|---|---|---|
| Уникальность идентификаторов | имя не должно быть объявлено повторно | повторное объявление `a` |
| Совместимость типов | справа от `=` должен быть кортеж | ожидался `Tuple` |
| Диапазон чисел | целые числа должны быть в диапазоне `0..2147483647` | слишком большое число |
| Использование идентификаторов | идентификатор внутри кортежа должен быть объявлен ранее | `b = {:bad, c};`, где `c` не объявлен |

## Примеры семантических ошибок

### Повторное объявление

```elixir
a = {:ok, 1};
a = {:user, 2};
```

Ожидаемая ошибка:

```text
идентификатор 'a' уже объявлен ранее (строка 1)
```

### Использование необъявленного идентификатора

```elixir
b = {:bad, c};
```

Ожидаемая ошибка:

```text
идентификатор 'c' используется до объявления
```

### Выход числа за допустимый диапазон

```elixir
big = {:n, 999999999999999999999};
```

Ожидаемая ошибка:

```text
число выходит за допустимый диапазон [0..2147483647]
```

# Ограничения реализации

- Поддерживается только учебная конструкция объявления кортежа, а не полный синтаксис языка Elixir.
- Идентификаторы распознаются в нижнем регистре, с цифрами и символом `_` после первого символа.
- В конце объявления требуется `;`, хотя в реальном Elixir такая запись не является обязательной.
- Область видимости для семантического анализа считается глобальной в пределах всего введённого текста.
- Строковые литералы поддерживают закрывающую кавычку и экранирование, но не являются полноценной реализацией всех строковых возможностей Elixir.
- Для сдачи работы желательно добавить реальные скриншоты всех тестов из запущенной программы.
